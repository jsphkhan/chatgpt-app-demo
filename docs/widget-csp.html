Content Security Policy (CSP)

How the sandboxed widget is allowed to load scripts and assets from your server

What is CSP?

Content Security Policy (CSP) is a browser security mechanism that restricts what a web page can load and connect to — which scripts, stylesheets, images, and network endpoints are permitted.

ChatGPT renders MCP App widgets inside a sandboxed iframe. By default that sandbox blocks all external origins. Your server must declare an allowlist so the widget can load its JS/CSS bundle and, if needed, call your APIs at runtime.

Where CSP is attached — widget.js

Before calling registerAppResource, the server builds a CSP object once and passes it in two places on the resource registration.

Code Description 
// widget.js
const csp = getWidgetCsp();

registerAppResource(
  server,
  "horoscope-widget",
  WIDGET_URI,
  { _meta: { ui: { csp } } },
  async () => ({
    contents: [
      {
        uri: WIDGET_URI,
        mimeType: RESOURCE_MIME_TYPE,
        text: widgetHtml,
        _meta: { ui: { csp } },
      },
    ],
  })
);

getWidgetCsp() (from config.js) returns the domain allowlist. The same csp object is attached:

  1. On the resource listing — the 4th argument to registerAppResource
  2. On the content item — inside the contents array returned by the callback

ChatGPT applies the content-level CSP when it fetches and renders the HTML. Both use the same object in this demo so listing and fetch stay in sync.

What are resourceDomains and connectDomains?

ChatGPT runs your widget in a sandboxed iframe with a restrictive Content Security Policy. The _meta.ui.csp object tells the host which external origins the widget is allowed to use. The two main fields control different kinds of access:

Field What it allows CSP mapping Example in this app
resourceDomains Loading static assets — JavaScript bundles, CSS, images, fonts, media referenced by <script>, <link>, <img>, etc. Maps to directives like script-src, style-src, img-src, font-src, media-src Required here. After loadWidgetHtml() rewrites paths, the iframe loads https://your-domain.ngrok.dev/assets/index-*.js and *.css. Without your server origin in resourceDomains, those requests are blocked and the widget stays blank.
connectDomains Runtime network requests from JavaScript — fetch(), XMLHttpRequest, WebSockets Maps to connect-src Not used by the horoscope widget today (data comes from ChatGPT via ontoolresult, not a direct API call). Included so the same CSP works if you later add client-side requests to your server (e.g. fetch("https://your-domain.ngrok.dev/api/…")).
Rule of thumbUse this field
Widget loads a JS/CSS file or image from a URL resourceDomains
Widget calls an API with fetch() or opens a WebSocket connectDomains
Same origin does both (e.g. API + static assets on one host) Add that origin to both arrays — as this demo does

Official reference: Apps SDK — _meta.ui.csp. There is also an optional frameDomains for embedding nested iframes; this horoscope app does not use it.

CSP on the tool as well

The same CSP object is passed into registerShowHoroscopeTool as widgetCsp and set on _meta.ui.csp alongside resourceUri. That tells ChatGPT the allowed domains before it fetches the widget HTML when the tool runs. See registerAppTool docs.
