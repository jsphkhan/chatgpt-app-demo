Publishing your App

Guidelines and steps for submitting a ChatGPT app for review

Overview

Publishing a ChatGPT app means submitting your MCP server and widget for OpenAI review so it can appear in the ChatGPT app directory. The ecosystem is built on trust — your app must be safe, useful, reliable, and respectful of user privacy.

Best practices for tools

MCP tools are the manual ChatGPT reads to use your app. Clear names, accurate descriptions, and correct annotations reduce review friction and help the model invoke tools safely.

Tool names

GuidelineExample
  • Unique within your app
  • Human-readable, verb-led, specific to the action
  • Avoid promotional or comparative words
 
// Good
show_horoscope
get_order_status
search_flights

// Avoid
best_horoscope
official_weather
pick_me

Tool descriptions

Code (this project)What makes it review-ready
 
description:
  "Displays a personalized horoscope card based on " +
  "the user's date of birth. When the user asks for " +
  "their horoscope, first ask for their date of birth " +
  "(YYYY-MM-DD), then call this tool with their DOB."
  • States what the tool does (displays a horoscope card)
  • Tells the model when to call it (user asks for horoscope)
  • Specifies required input format (YYYY-MM-DD)
  • Does not disparage other apps or ask the model to prefer this tool
  • Does not trigger on overly broad intents unrelated to horoscopes

Tool annotations

Annotations tell ChatGPT and users how cautious to be. Incorrect labels are a common rejection reason — you must justify each annotation at submission time.

Annotation Meaning When to use
readOnlyHint Tool only reads or displays data; does not change anything outside ChatGPT Lookups, search, status checks, read-only generators (like this demo’s horoscope)
destructiveHint Tool performs irreversible or high-impact writes (delete, cancel, overwrite) Deleting records, canceling orders, removing accounts
openWorldHint Tool interacts with external systems, accounts, or public platforms Posting to social media, sending email, charging a card, publishing public content
ExamplesTypical annotations
 
// Read-only lookup (horoscope demo)
show_horoscope
  readOnlyHint: true
  destructiveHint: false
  openWorldHint: false

// Write to external system
send_email
  readOnlyHint: false
  destructiveHint: false
  openWorldHint: true

// Destructive external action
delete_user_account
  readOnlyHint: false
  destructiveHint: true
  openWorldHint: true

show_horoscope in this demo only generates and returns data for the widget — it does not post, charge, or modify external state. That maps to read-only, non-destructive, closed-world.

Any tool that sends data outside the chat (email, posts, uploads) should set openWorldHint: true and usually readOnlyHint: false. Destructive actions need destructiveHint: true plus clear description text so clients can show confirmation.

Also keep inputs minimal and purpose-driven — request only fields required for the task (e.g. dob for a horoscope, not full chat history or precise GPS).
